

# UX Vern's Analysis: Dark-Themed Todo + Kanban App

*Puts on user empathy goggles. Takes off developer hat. Burns developer hat.*

---

## First Question: Who Is This Person?

Before I touch a single component, let me paint the picture. Our user is someone who:

- Has tried Trello, Todoist, Notion, and six other apps this year
- Wants something **simple enough to not feel like work** but **powerful enough to actually organize their life**
- Opens this app at 8am with coffee, or at 11pm on their phone when they remember something
- Does NOT care about your data model, your normalized store, or your order field strategy
- Will abandon this app in 48 hours if the first 5 minutes don't click

MightyVern did great work on the architecture. Truly. But I need to challenge some things because **architecture doesn't matter if the user bounces on day one.**

---

## The First 5 Minutes — This Is Where You Win or Lose

The plan jumps straight to data models and component hierarchies. But here's what the user actually experiences:

1. **They land on the app.** What do they see?
2. **It's empty.** Every column is blank. Every list is bare. This is the scariest moment.
3. **They need to figure out how to add a task.** Is it obvious? Or do they stare at an empty board wondering where to click?

### What's Missing: Empty States

The plan mentions "empty state UI" as **medium priority**. No. This is **critical priority**. The empty state IS your onboarding. Here's what I want:

- **Board view, first visit:** Columns pre-populated with defaults (e.g., "To Do," "In Progress," "Done"). A friendly placeholder card in the first column that says something like *"Click + to add your first task"* — not a developer tooltip, an actual visual invitation.
- **List view, first visit:** A centered message. Not "No items found" (that's a database talking). Something like *"Nothing here yet. Add your first task above."* with an arrow or visual cue pointing to the input.
- **Detail panel, empty:** When no task is selected, show a hint: *"Select a task to see its details"* — don't show a blank white (or in this case, blank dark) void.

**The user sees:** An empty screen.
**The user expects:** Guidance on what to do next.
**The user gets (if we don't fix this):** Confusion and a closed tab.

## The View Toggle: A Confusion Trap

The plan describes two views — board and list — with a toggle that persists. Here's my concern:

### Who uses which view and when?

- **Board view** = visual thinkers, project planners, people who like spatial organization
- **List view** = checklist people, "just tell me what's next" people, mobile users

These are potentially **different users with different mental models**. The toggle needs to be:

1. **Visible but not confusing.** Don't hide it in a settings menu. Don't make it a tiny icon that looks like a layout bug.
2. **Labeled clearly.** Not just icons — use text labels or at minimum, tooltips that say "Board View" and "List View," not "Grid" and "Lines."
3. **Non-destructive.** The plan says "switching views does not discard selection or edits in progress" — good, but let me push harder: if I'm editing a task detail panel and I switch views, **keep that panel open.** Don't close it. Don't make me find my task again.

**The user sees:** Two mysterious icons in the toolbar.
**The user expects:** To understand what they do without clicking them.
**The user gets (if we don't fix this):** Anxiety about clicking the wrong thing and losing their place.

## Task Creation: The Most Important 3 Seconds

How does a user add a task? The plan describes Task CRUD but doesn't specify the interaction pattern. This is THE most critical UX decision.

### My recommendation: Quick capture first, details later

1. **Board view:** A visible "+ Add task" button at the bottom (or top) of each column. One click opens an inline text field. Type a title, press Enter. Done. Task created. Details come later.
2. **List view:** A persistent input at the top of the list. Always visible. Type, Enter, done.
3. **Detail comes after:** Click the newly created task to open the detail panel for markdown, tags, dates.

### What NOT to do:
- Don't open a modal with 8 fields for a new task. That's a form, not a task capture.
- Don't require a title AND a description AND a status before saving. Minimum viable task = title only.
- Don't make the user pick a column/status before creating. Default to "To Do" or the first column.

**Progressive disclosure is everything here.** Capture the thought fast. Enrich it later.

## The Markdown Editor: Cool Feature, UX Minefield

I love the ambition. Markdown is powerful. But let me be real:

### Most users don't know markdown.

The plan assumes a user who writes `## Heading` and `**bold**`. That's a developer. The actual user will type plain text and wonder why their asterisks aren't disappearing.

### My recommendations:

1. **Toolbar is mandatory, not optional.** Bold, italic, heading, list, checkbox, link. Clickable buttons. Always visible.
2. **WYSIWYG-ish or live preview.** The plan mentions split view on desktop, toggle on mobile. Split view is fine for desktop, but consider: a live-rendering editor (like Notion or Typora) where the markdown renders as you type would be far more intuitive than a raw-text-on-left, preview-on-right split.
3. **Don't call it a "markdown editor."** Users don't care about the format. Call it "Notes" or "Details." The word "markdown" is developer jargon.
4. **The empty state matters here too.** When the markdown body is empty, show placeholder text: *"Add notes, checklists, links..."* — not a blank textarea.

**The user sees:** A text area.
**The user expects:** To type and have it look nice.
**The user gets (if we don't fix this):** Raw markdown syntax they don't understand.

---

## Tags: Powerful but Potentially Overwhelming

The plan has a full tag management system — create, rename, recolor, delete. That's a lot of UI surface for something that should feel lightweight.

### Simplify the happy path:

1. **Creating a tag should happen inline during task tagging.** Type a tag name, it doesn't exist yet? Offer to create it right there. Don't make me go to a "Tag Manager."
2. **Auto-assign a color.** Don't make me pick one. Give me a nice default from a curated palette. Let me change it later if I care (most won't).
3. **Tag selection in the detail panel:** Show existing tags as clickable chips. Show a text input for search/create. That's it.
4. **Tag deletion:** Only accessible from a settings area, not inline. Accidental tag deletion is catastrophic if it removes the tag from 50 tasks.

### Color contrast — the plan mentions this, but let me be specific:

On a dark background, light-colored tag chips with dark text work. Dark-colored tag chips with light text also work. What DOESN'T work: medium-saturation colors that look muddy against dark gray. **Pre-curate 8-12 tag colors that are tested against your dark theme.** Don't give users a full color picker — they'll pick something illegible every time.

## Date Pickers: Where Users Go to Die

Date pickers are one of the most hated UI elements in existence. The plan mentions "calendar pickers" without specifying behavior.

### Keep it simple:

1. **Click the date area → calendar dropdown appears.** Not a modal. Not a new page. A popover.
2. **Quick shortcuts matter.** "Today," "Tomorrow," "Next Week" buttons above the calendar. Most tasks don't need a specific date hunt.
3. **Clear the date easily.** A visible "Remove date" option. Don't trap me into always having a date.
4. **Due date visual urgency** — the plan's badge system is good, but be careful with color: don't use red for "due today" because red means "overdue/danger." Use amber/yellow for "due soon," red for "overdue."
5. **Start date is a power-user feature.** Hide it behind a "Add start date" link. Most tasks don't need one. Don't show two date fields by default.

---

## Drag and Drop: The "Looks Easy, Breaks Everything" Feature

The plan covers the technical risks well. Let me add the UX risks:

1. **Discoverability:** How does the user know they CAN drag? Add a subtle grip handle (⠿) on the left side of cards. Don't rely on "the whole card is draggable" — that conflicts with clicking to open details.
2. **Visual feedback during drag:** The card should lift (shadow, slight scale), the drop zone should highlight. Without this, dragging feels broken.
3. **Mobile:** Drag-and-drop on mobile is rough. The plan mentions providing a "non-drag alternative." I'll be more specific: add a **status dropdown** in the detail panel and/or on long-press. Let mobile users CHANGE status without dragging. Dragging on a phone-sized kanban board is painful.
4. **Undo:** If I accidentally drop a task in the wrong column, can I undo? A simple toast notification with "Undo" link is trivial to implement and saves enormous frustration.

---

## Responsive / Mobile: Not an Afterthought

The plan mentions mobile in passing. Let me be direct:

### On mobile, the kanban board doesn't work well.

Horizontal scrolling across columns on a phone is awkward. Three columns don't fit. Consider:

1. **Default to list view on mobile.** Auto-switch based on viewport, with the option to toggle.
2. **If showing the board on mobile:** Show one column at a time with swipe navigation between columns. Like Trello mobile.
3. **The detail panel on mobile** should be a full-screen slide-over, not a side panel. There's no room for a side panel on a 375px screen.
4. **The markdown editor on mobile** should be toggle (edit/preview), not split. There's no room for split view.

## Error States & Edge Cases: The Empathy Checklist

The plan lists 10 edge cases. Good. But what does the USER experience?

| Situation | What the user sees | What they should see |
|---|---|---|
| Task deleted while detail panel is open | ??? | Panel closes gracefully with a brief toast: "Task was deleted" |
| Tag deleted that's on tasks | ??? | Tag disappears from tasks silently. No error. No orphaned references. |
| Start date after due date | A warning | A gentle inline hint: "Start date is after due date — is that right?" Not a blocking error. |
| localStorage full | ??? | A non-blocking banner: "Storage is getting full. Consider cleaning up completed tasks." NOT a crash. |
| Overdue task marked done | ??? | Badge changes to "Completed" — don't shame them with a red overdue badge on a done task |
| Pasting rich text into markdown | Garbled formatting | Clean paste. Strip HTML or convert gracefully. |

## What I'd Reprioritize from the Plan

### Move UP in priority:
- **Empty states** (currently medium → should be P0)
- **Quick task capture UX** (not specified → should be P0)
- **Mobile-responsive layout** (barely mentioned → should be P1)
- **Undo for drag-and-drop** (not mentioned → should be P1)

### Move DOWN in priority:
- **`shortDescription` as a separate field** — This adds cognitive load. Most users won't fill in both a title AND a short description AND markdown notes. Consider: title + markdown body only. The "short description" can be auto-derived from the first line of markdown, or just be the title itself on cards.
- **`startDate`** — Power user feature. Ship without it. Add later behind progressive disclosure.
- **Tag rename/recolor management** — Ship with auto-colors and inline creation. Full management can come in v2.

## The Open Questions — My Answers

1. **Multiple boards or single board?** Single board. Multiple boards = multiple places to lose tasks = cognitive overload. Add boards later if users ask.
2. **Completion: boolean, status, or both?** Status-based, but the user should think in terms of a checkbox. Checking "done" moves it to the Done column. Unambiguous.
3. **Priorities beyond tags?** Not yet. Tags + due dates cover 90% of urgency. Priority flags add another thing to learn.
4. **Collaboration/sharing?** Not in v1. localStorage means single user. Design for one person's brain.
5. **Markdown checklists syncing with completion?** No. That's clever engineering but confusing UX. Is the task done when the checkbox is checked, or when all the sub-checkboxes are checked? Pick one mental model.

---

## Summary: The UX Non-Negotiables

1. **Empty states are onboarding.** Design them first, not last.
2. **Task creation must be instant.** Title → Enter → Done. Details come later.
3. **Don't say "markdown."** Call it "Notes." Add a toolbar.
4. **Curate tag colors.** Don't hand users a full color picker on a dark theme.
5. **Mobile is a different app.** Design it as one, not as a shrunken desktop.
6. **Drag-and-drop needs visible handles, clear feedback, and an undo toast.**
7. **Kill `shortDescription` as a separate field.** Title + notes is enough.
8. **Ship with generous defaults.** Pre-made columns, auto-assigned tag colors, default to list view on mobile.

The architecture in the plan is solid. The data model is clean. But none of that matters if the user opens the app, sees an empty dark screen with no guidance, and closes the tab.

**Build for the person. Not the schema.**
